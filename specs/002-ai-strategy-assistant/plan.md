# Implementation Plan: AI Strategy Assistant

**Branch**: `002-ai-strategy-assistant` | **Date**: 2026-04-16 | **Spec**: [spec.md](./spec.md)
**Input**: Feature specification from `/specs/002-ai-strategy-assistant/spec.md`

## Summary

Adds an AI Strategy Assistant sidebar and (stretch) a Workload Heatmap to the existing Caseload Overview route (`/`). Both are presentation layers over Stream A's existing derivations — they introduce no new routes, no new persistence, and no new derivation primitives.

The deterministic pieces (triage counts, priority-case selection, recommended action, fallback sentence) are pure functions of the already-computed `EnrichedCase[]`. Only the human-readable insight sentence is generated by a live OpenAI Chat Completions call at render time, supplied with the case reference, applicable policy identifier, top risk factors, and recommended next-action label so the model's output names the same artefacts the deterministic selection chose. Three failure modes (no API key, call fails/times out, malformed response) each degrade to a deterministic fallback sentence so the demo path always renders.

Scope: one new stream of work (call it Stream F) that depends only on Stream A and Stream C — no changes required to Streams B or D.

## Technical Context

**Language/Version**: TypeScript 5.8 (strict), Node ≥ 18 for tooling — unchanged from 001.
**Primary Dependencies**: React 18.3, React Router DOM 6.30, Vite 5.4, Tailwind 3.4, shadcn-ui, `lucide-react` — unchanged. No OpenAI SDK; use `fetch` directly against `https://api.openai.com/v1/chat/completions` to keep bundle size and review surface minimal (per Constitution §IV).
**Storage**: No change. Static JSON fixtures only. The assistant reads `EnrichedCase[]` from `services/cases.ts`; it writes nothing.
**Testing**: Vitest — one new test file (`src/test/assistant.test.ts`) covering the deterministic-selection rule (FR-104), fallback-sentence composer (FR-119c), and no-key/malformed-response empty states (FR-119b). The live OpenAI endpoint is mocked via `vi.stubGlobal('fetch', …)` — no test calls a real API.
**Target Platform**: Same as 001 — modern desktop browser, 1440×900 reference viewport.
**Project Type**: Single-project SPA (Vite + React). No change.
**Performance Goals**: Deterministic selection + triage counts render synchronously on filter change (< 16 ms per tick, i.e. one frame). The OpenAI call is async with a 5-second `AbortController` timeout; during the pending window the insight bubble shows a skeleton, the rest of the panel (triage counts + CTA) is already interactive.
**Constraints**:
- No write operations on any route (unchanged).
- No global state library (unchanged).
- Frozen reference date for all derivations (unchanged).
- OpenAI API key MUST be supplied via `VITE_OPENAI_API_KEY` at build time. It is NOT committed. When absent, the panel degrades per FR-119b.
- OpenAI call uses `gpt-4o-mini`, `temperature: 0.2`, `max_tokens: 180` — chosen for latency, cost, and reproducibility. No streaming.
- The key is embedded in the built bundle (Vite inlines `VITE_*` vars at build time); this is documented as a hackathon-only trade-off, not production practice. See research.md §4.

**Scale/Scope**: Same 50-row soft cap as 001. One outbound API call per re-render of the assistant (throttled by React memoisation on selected-case identity — no call if the same case remains the priority across filter changes).
